# POS 外部串接 Roadmap

## 目標

這份文件整理 NUCOSMOS POS 前台未來可串接的外部系統與設備，並依照實務導入優先順序分階段規劃，方便後續施工、測試與對外採購。

## 核心原則

- 先做門市現場最常用的設備與服務
- 優先選擇有正式 API 或 Android SDK 的廠商
- 能走標準 HTTP / Android intent / 藍牙的方案優先
- 串接前先確認 POS 前台、後端、管理後台三端責任分工

## 第一階段：開店必備

### 1. 熱感出單機 / 小白單機

- 目的：列印廚房單、客單、交班單、退款單
- 常見串接方式：
  - Android 廠商 SDK
  - 藍牙列印
  - USB / Serial
  - ESC/POS 指令集
- 建議能力：
  - 測試列印
  - 依門市 / 區域 / 品項分流列印
  - 補印與重印
  - 廚房單 / 顧客單不同版型

### 2. 錢櫃

- 目的：現金結帳完成後自動彈開
- 常見串接方式：
  - 出單機 RJ11 連動
  - Android 外接裝置 SDK
- 建議能力：
  - 僅限現金交易可開啟
  - 操作記錄留痕

### 3. 條碼掃描器

- 目的：掃會員碼、商品條碼、優惠券條碼
- 常見串接方式：
  - USB HID 鍵盤模式
  - 藍牙 HID
  - Android 專用 SDK
- 建議能力：
  - 掃描商品加入購物車
  - 掃描會員 / 票券 / 訂單條碼

### 4. 第三方支付 / 刷卡

- 目的：刷卡、行動支付、聚合支付
- 常見串接方式：
  - Android 廠商 SDK
  - WebView 支付頁
  - Server-to-server payment API
- 建議能力：
  - 現金 / 信用卡 / QR Code 分流
  - 支付成功回寫訂單
  - 取消與退款
  - 支付交易號保存

### 5. 電子發票

- 目的：開立發票、作廢、折讓
- 常見串接方式：
  - 雲端發票服務商 API
  - 第三方 SaaS 平台
- 建議能力：
  - 訂單成立後自動開票
  - 載具 / 統編 / 捐贈碼輸入
  - 退款後作廢或折讓

## 第二階段：營運與會員

### 1. 會員系統

- 目的：累點、儲值、會員價、生日禮
- 常見串接方式：
  - 自建會員 API
  - 第三方 CRM / CDP
- 建議能力：
  - 手機號碼 / 條碼 / QR Code 查會員
  - 點數累積與折抵
  - 會員專屬活動價

### 2. LINE / 社群會員

- 目的：會員綁定、推播、點數整合
- 常見串接方式：
  - LINE Login
  - LIFF
  - LINE Messaging API
- 建議能力：
  - LINE 綁定會員
  - 領券 / 集點 / 通知

### 3. 優惠券與折扣碼

- 目的：行銷活動與現場折扣核銷
- 常見串接方式：
  - 自建 coupon API
  - 第三方票券平台 API
- 建議能力：
  - 掃碼核銷
  - 使用條件驗證
  - 單次 / 多次 / 時段限制

### 4. 簡訊驗證

- 目的：會員綁定、 OTP 登入、驗證身分
- 常見串接方式：
  - SMS API
- 建議能力：
  - 發送 OTP
  - 短時間重送限制

## 第三階段：通路與營運整合

### 1. 外送平台

- 目的：整合平台訂單
- 常見串接方式：
  - Food delivery partner API
  - 中介聚合平台
- 建議能力：
  - 訂單同步
  - 狀態回寫
  - 品項對碼

### 2. 廚房顯示系統 KDS

- 目的：出餐看板、製作狀態管理
- 常見串接方式：
  - WebSocket
  - HTTP polling
  - Android TV / 平板應用
- 建議能力：
  - 新單提示
  - 製作中 / 完成狀態
  - 分站台顯示

### 3. 會計系統 / ERP

- 目的：對帳、總帳、採購與庫存整合
- 常見串接方式：
  - API
  - CSV 匯出匯入
- 建議能力：
  - 銷售摘要匯出
  - 發票資料對接
  - 採購與庫存同步

### 4. BI / 報表平台

- 目的：進階分析與管理視圖
- 常見串接方式：
  - ETL
  - API / Data warehouse
- 建議能力：
  - 銷售趨勢
  - 毛利分析
  - 門市比較

## 技術建議

### Flutter POS 前台

- 優先支援：
  - HTTP API
  - Android SDK
  - Bluetooth / USB
  - Android intent
- 建議設計：
  - 每種外部設備抽成 adapter / service
  - 裝置設定由管理後台集中管理
  - POS 端保留測試連線與設備診斷頁

### Spring Boot 後端

- 負責：
  - 訂單主流程
  - 支付結果落帳
  - 發票資料保存
  - 會員、券、點數邏輯
- 建議：
  - 第三方交易號與 request/response 都保留審計記錄
  - 支付與發票流程做 idempotency 保護

### 管理後台

- 建議新增：
  - 外部串接設定
  - 支付方式管理
  - 發票設定
  - 列印機與設備管理
  - 串接狀態監控

## 最建議的施工順序

1. 熱感出單機
2. 第三方支付
3. 電子發票
4. 會員系統
5. 條碼掃描器
6. 外送平台

## 現階段建議

如果目標是讓 POS 盡快進入可用測試，最推薦先做這三項：

1. 熱感出單機
2. 第三方支付
3. 電子發票

這三項完成後，POS 前台就會從「可點餐測試」進一步變成「接近可營運版本」。
