# POS 系統實用架構 PRD

## 1. 文件資訊

- 文件名稱：POS 系統實用架構與產品需求文件
- 專案名稱：NUCOSMOS POS
- 文件版本：v0.1
- 建立日期：2026-03-20
- 適用範圍：安卓平板 POS 前台、Web 後台管理、後端 API 與資料庫、離線同步機制

## 2. 背景與目標

本專案目標是建立一套可部署於大陸製安卓平板的 POS 系統，支援門市前台收銀、商品管理、訂單管理、報表查詢與後續擴充。由於平板設備可能存在硬體規格不一致、ROM 客製、瀏覽器能力差異、網路不穩與周邊設備整合需求，因此系統應以穩定營運為第一優先，而不是只追求前端開發速度。

本系統建議採用以下技術方向：

- POS 前台：Flutter Android App
- 後台管理：Vue Web
- 後端服務：Backend API Service
- 資料庫：集中式主資料庫
- POS 本機：本地資料庫與同步模組

## 3. 產品目標

### 3.1 業務目標

- 支援單店與多店營運
- 支援多台 POS 同步商品、價格、訂單與庫存資料
- 支援店員日常收銀流程，縮短點單與結帳時間
- 提供後台管理能力，讓非工程人員可維護商品、分類、促銷與報表
- 在網路不穩時仍可持續營運

### 3.2 系統目標

- 前台 POS 可在離線狀態下完成基本銷售流程
- 資料在恢復連線後自動與伺服器同步
- 支援藍牙或 USB 印表機、掃碼器等外設整合
- 支援未來擴充會員、優惠券、庫存、交班、發票或電子支付
- 提供可審計、可追蹤、可維護的後端資料架構

## 4. 非目標

以下項目不列為第一階段必做範圍：

- 跨國稅制與多國語系
- 複雜 ERP 整合
- AI 推薦與進階會員分群
- 極完整的會計模組
- 所有支付方式的一次性整合

## 5. 目標使用者

### 5.1 前台使用者

- 店員
- 店長
- 門市主管

### 5.2 後台使用者

- 總部營運人員
- 商品管理人員
- 財務或對帳人員
- 系統管理員

## 6. 使用情境

### 6.1 門市 POS 前台

- 店員在安卓平板上點選商品、調整規格、加入購物車
- 建立訂單並完成結帳
- 列印小票或廚房單
- 網路中斷時持續收銀
- 網路恢復後補同步

### 6.2 後台管理

- 維護商品、分類、價格、門市可售狀態
- 管理訂單、退款、作廢與日結資訊
- 管理員工帳號與權限
- 查詢營業報表、商品銷售排行、時段分析

## 7. 建議整體架構

### 7.1 架構結論

本系統建議採用「前後分離 + 離線優先」架構：

- Flutter App 負責門市 POS 前台
- Vue Web 負責後台管理
- Backend API 負責業務規則、認證授權、同步與整合
- 主資料庫作為唯一可信資料來源
- POS 本機保留可離線操作的必要資料與未同步交易

### 7.2 架構示意

```text
[Vue 後台管理]
        |
        v
[Backend API / Auth / Sync Service]
        |
        v
[Main Database]
        ^
        |
[Flutter POS App] <-> [Local Database on Tablet]
        |
        +-- Bluetooth / USB Printer
        +-- Scanner
        +-- Optional NFC / Payment Device
```

### 7.3 為什麼不建議把前後台和資料都做在 POS 前台

- 資料無法集中管理
- 多台設備難以同步
- 壞機或換機時有資料遺失風險
- 後台管理與報表難以建立
- 權限與審計紀錄難以落實
- 商業邏輯散落在裝置端，後續維護成本高

## 8. 技術選型建議

### 8.1 POS 前台

- 建議：Flutter
- 原因：
  - 對 Android 裝置支援更穩定
  - 較容易整合藍牙、USB、列印、掃碼等硬體
  - 較容易做 kiosk 模式、開機自啟與裝置管控
  - 不依賴平板瀏覽器版本與 WebView 能力

### 8.2 後台管理

- 建議：Vue 3
- 原因：
  - 適合表單、表格、查詢、權限與設定頁
  - Web 元件生態成熟
  - 適合桌機瀏覽器操作
  - 後續擴充管理功能成本較低

### 8.3 後端服務

- 建議：Java Spring Boot
- 原則：
  - 提供 REST API
  - 保留未來拆分服務的空間
  - 將商業規則集中在後端

### 8.4 資料庫

- 建議主資料庫：PostgreSQL
- POS 本地資料庫：SQLite

## 10. 核心資料邊界

### 10.1 必須以伺服器為主資料來源的資料

- 商品主檔
- 商品分類
- 價格規則
- 優惠規則
- 員工帳號與權限
- 門市資料
- 訂單主檔
- 退款與作廢紀錄
- 報表統計資料

### 10.2 必須存在 POS 本機的資料

- 當前門市可售商品快取
- 當前價格與分類快取
- 本機登入狀態
- 未同步訂單
- 未同步退款或作廢操作
- 列印任務佇列
- 裝置設定

### 10.3 同步原則

- 商品、分類、價格等主資料由伺服器下發至 POS
- POS 訂單先寫入本機，再同步到伺服器
- 每筆訂單需有全域唯一 ID
- 每筆同步事件需有時間戳與裝置識別
- 同步衝突以伺服器規則為準，但需保留人工介入能力

## 11. 離線優先設計

### 11.1 設計原則

- 沒網路時仍能開單與結帳
- 前台操作不依賴即時 API 回應
- 同步失敗時可重試，不影響店員持續收銀
- 重要交易資料不可只存在記憶體，必須立即落本機資料庫

### 11.2 離線可用功能

- 瀏覽商品
- 加入購物車
- 建立訂單
- 現金結帳
- 列印小票
- 查詢當前班次的本機訂單

### 11.3 離線限制

- 即時跨店庫存一致性無法保證
- 雲端會員點數可能延後同步
- 第三方支付狀態可能需要待網路恢復後確認

## 12. 同步機制需求

### 12.1 下行同步

伺服器同步至 POS 的資料：

- 商品
- 分類
- 價格
- 促銷
- 門市設定
- 裝置授權狀態

### 12.2 上行同步

POS 同步至伺服器的資料：

- 訂單
- 訂單明細
- 退款
- 作廢
- 班次資訊
- 裝置心跳與狀態

### 12.3 同步策略

- App 啟動時同步
- 登入後同步
- 固定排程同步
- 手動觸發同步
- 網路恢復時立即同步

### 12.4 同步狀態 UI

POS 前台需顯示：

- 在線或離線狀態
- 最後同步時間
- 待同步筆數
- 同步失敗告警

## 13. 硬體整合需求

### 13.1 必要外設

- 藍牙或 USB 小票印表機
- 掃碼器

### 13.2 可選外設

- 錢櫃
- NFC 讀卡裝置
- 客顯裝置
- 電子秤

### 13.3 設計要求

- 外設整合盡量透過原生能力處理
- 列印失敗需可重試
- 外設狀態需能在裝置管理中查看

## 14. 安全與權限

### 14.1 認證

- POS 使用裝置綁定 + 員工登入
- 後台使用帳號密碼登入，可後續擴充雙重驗證

### 14.2 授權

- 以角色為基礎的權限模型
- 區分總部、店長、店員、財務、系統管理員

### 14.3 安全要求

- API 全程 HTTPS
- Token 應具有效期與刷新機制
- 敏感操作需記錄審計日誌
- 本機資料庫中的敏感資料應避免明文保存

## 16. 非功能需求

### 16.1 效能

- POS 主要操作應在 300ms 內得到 UI 回應
- 商品列表在本機快取情境下應快速開啟
- 結帳流程不得因同步阻塞 UI

### 16.2 可用性

- POS 前台介面需針對平板觸控最佳化
- 大按鈕、大字體、低誤觸率
- 關鍵操作需有明確成功與失敗提示

### 16.3 穩定性

- App 異常重啟後應可恢復未完成同步資料
- 同步模組需支援重試與錯誤追蹤

### 16.4 可維護性

- 前後台與後端分層清楚
- 關鍵商業規則集中於後端
- API 與資料模型需可版本化

## 18. API 規劃原則

- 採 REST API 為主
- API 路徑明確版本化，例如 `/api/v1/...`
- 寫操作需具備冪等性設計
- 訂單上傳應支援批次與重試
- 裝置心跳與同步狀態需可查詢

## 21. 驗收標準

### 21.1 POS 前台

- 無網路時可完成現金訂單建立
- 網路恢復後訂單可成功同步
- 商品資料可由後台更新後同步至平板
- 小票可穩定列印

### 21.2 後台

- 可新增、修改、停用商品
- 可查詢訂單與營收
- 可管理員工與裝置

### 21.3 後端

- API 可穩定支援多台 POS 同步
- 主資料與交易資料可追蹤
- 具備基本權限控管與審計能力

## 22. 主要風險

- 大陸製平板硬體能力差異大，外設相容性需先驗證
- 若一開始不做離線設計，後期重構成本會很高
- 若商業規則先寫死在 POS 端，後續維護成本會快速上升
- 若沒有裝置管理與同步監控，營運問題難以排查

## 23. 開發建議結論

本案建議採用以下組合：

- POS 前台：Flutter
- 後台管理：Vue 3
- 後端：Java Spring Boot API Service
- 主資料庫：PostgreSQL
- POS 本機資料庫：SQLite
- 架構策略：前後分離、離線優先、同步驅動

此架構兼顧：

- 安卓平板部署穩定性
- 後台管理開發效率
- 多裝置同步能力
- 後續營運擴充性
- 硬體整合可行性
